# ASCII日付パース高速化と暦定義deep freeze実装計画

**目的**：和暦拡張を読み込んだ後も、ASCIIだけで構成された通常の日付文字列を最小の追加コストで標準パーサーへ渡す。
同時に、旧暦変換が参照する年定義と内部配列を凍結し、公開定数の変更によるプロセス全体の変換破損を防ぐ。

**対象**：包括レビューの所見4と所見7だけを修正する。
通常の `strftime`、元号探索、暦テーブルの遅延ロードは対象外とする。

**互換性条件**：Ruby 2.0で解釈できる構文だけを使う。
ASCII以外を含む和暦、日本語時刻、月の別名、標準パーサーへの寛容なフォールバックは維持する。

**実施状態**：2026-07-14に実装と独立レビューを完了した。
最終検証はRSpec 92件成功、RuboCop 17ファイル違反なし、Ruby構文検査成功である。
ASCIIの `Date.parse` は約2.8 µsから約2.2 µsへ短縮し、保存済み標準パーサーとの差は約0.01 µsまで縮小した。
deep freezeの追加起動時間は約0.07 msで、全1,428件の年定義と内部配列の凍結を確認した。

## Task 1：ASCII入力を標準日付パーサーへ直行させる

**難易度**：高。
グローバルに差し替えた `Date.parse` と `Date._parse` の呼び出し頻度が高いため、性能値だけでなく和暦経路を迂回しない境界も確認する。

**変更対象**：

- `lib/wareki/std_ext.rb`
- `spec/std_ext_spec.rb`

- [ ] 変更前の `Date.parse('2025-07-01')` と `Date._parse('2025-07-01')` について、元実装との時間差と割り当て数を同一スクリプトで記録する。
- [ ] ASCII入力で `Wareki::Utils.normalize_time` を呼ばないことを示す失敗テストを追加する。
- [ ] `Date.parse` と `Date._parse` の冒頭で入力の `to_s.ascii_only?` を確認し、該当時は元実装へ返す。
- [ ] ASCIIのISO日付、英語日付、ASCII時刻が標準実装と同じ結果になることを確認する。
- [ ] 和暦、日本語時刻、弥生、全角数字を含む既存入力が従来の経路で処理されることを確認する。
- [ ] quick filter一致後の `ArgumentError` から標準パーサーへ戻る経路を維持する。
- [ ] 変更後に同じベンチマークを実行し、時間と割り当て数を比較する。
- [ ] 対象specと全spec、RuboCopを実行する。

**完了条件**：ASCII入力では和暦用の正規表現と時刻正規化を実行せず、和暦と日本語時刻の全既存テストが成功する。

## Task 2：年定義と内部配列をdeep freezeする

**難易度**：中。
生成済みデータだけを修正すると再生成時に問題が戻るため、生成スクリプトと出力ファイルを同時に変更する。

**変更対象**：

- `build-util/gen-jp-cal-def.rb`
- `lib/wareki/calendar_def.rb`
- `spec/utils_spec.rb`

- [ ] `YEAR_DEFS` の各 `Year`、`month_starts`、`month_days` が凍結済みであることを示す失敗テストを追加する。
- [ ] 各配列の破壊的変更と `Year` の属性変更が `RuntimeError` を送出することを確認する。
- [ ] `calendar_def.rb` の定義末尾で、内部配列、`Year`、外側配列の順に凍結する。
- [ ] 生成スクリプトも同じdeep freezeコードを出力するよう変更する。
- [ ] `YEAR_BY_NUM` が同じ凍結済みオブジェクトを参照することを確認する。
- [ ] 代表的な旧暦の双方向変換を実行し、読み取り処理に回帰がないことを確認する。
- [ ] 対象specと全spec、RuboCop、Ruby構文検査を実行する。

**完了条件**：公開された年定義をどの階層からも変更できず、生成処理を再実行しても同じ不変条件が保たれる。
